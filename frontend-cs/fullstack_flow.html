브라우저 입력 → 네트워크 → Spring 서버 → DB → 응답 → 렌더링 (HTML / React 비교)

→
STEP 2 브라우저 캐시 확인 있으면 바로 응답

→
STEP 5 TCP 3-way Handshake SYN → SYN-ACK → ACK
→
STEP 6 TLS/HTTPS 연결 암호화 채널 수립
→
STEP 7 Request 전송 패킷으로 분할 전송

STEP 8 DispatcherServlet 요청 진입점
→
STEP 9 Controller @GetMapping("/search")
→
STEP 10 Service 비즈니스 로직
→
STEP 11 Repository DB 쿼리

STEP 12 SQL 실행 SELECT * FROM ...
→
STEP 13 결과 반환 ResultSet → Entity

→

브라우저 캐시 확인 순서
1
Memory Cache 확인
현재 탭의 메모리에 있는 캐시. 가장 빠름. 탭 닫으면 사라짐.
~0ms
2
Disk Cache 확인
하드디스크에 저장된 캐시 (이전 방문 기록). 재시작해도 남아있음.
~수ms
3
캐시 없음 → 실제 네트워크 요청
처음 방문이거나 캐시가 만료되면, DNS → TCP → HTTP 요청 전체 수행
수십~수백ms
DNS Lookup — 도메인을 IP로 바꾸는 과정
DNS 
/* 브라우저는 www.google.com의 IP를 모름 → DNS에게 물어봄 */

브라우저 → 로컬 hosts 파일 확인
       → 로컬 DNS 캐시 확인
       → 공유기(로컬 DNS 서버) 확인
       → ISP DNS 서버 확인
       → 루트 DNS 서버(.com 담당)
       → 구글 DNS 서버
       ← 142.250.196.100 반환  ✅

/* 최초 1회만 수행, 이후 TTL 동안 캐시에 저장 */
google.com → 142.250.196.100  (TTL: 300초)
TCP 3-way Handshake + HTTPS 수립
TCP / TLS 
/* TCP 연결 수립 (신뢰성 있는 통신 채널 만들기) */

브라우저  →  SYN          →  서버
브라우저  ←  SYN-ACK     ←  서버
브라우저  →  ACK          →  서버
                    ↑ 여기까지 TCP 연결 완료

/* HTTPS = TLS 암호화 추가 */
브라우저  →  ClientHello (지원 암호화 목록)
서버      ←  ServerHello + 인증서(SSL Certificate)
브라우저     인증서 검증 (CA 신뢰 확인)
브라우저  →  대칭키 교환 (암호화)
                    ↑ 이제 모든 통신이 암호화됨 🔒

/* 총 소요: 약 50~200ms (지역마다 다름) */
실제 요청 전송 과정
→
Request 패킷 분할 전송
HTTP 요청이 여러 개의 TCP 패킷으로 분할되어 인터넷을 통해 전송됩니다.
각 패킷은 서로 다른 경로로 이동할 수도 있습니다.
↓
서버의 Load Balancer 도달
실제 대형 서비스는 수천 대의 서버가 있고, Load Balancer가 요청을 분산합니다.
구글 기준: 전 세계 데이터센터 중 가장 가까운 곳으로 라우팅
↑
Spring 서버 수신
DispatcherServlet이 요청을 받아서 처리 시작
Spring MVC 내부 처리 흐름
Spring Java 
// ① DispatcherServlet이 모든 요청을 받는 진입점
// GET /search?q=claude → 여기로 들어옴

// ② Controller — URL 매핑 & 파라미터 추출
@RestController
public class SearchController {

    @Autowired
    private SearchService searchService;

    @GetMapping("/search")
    public ResponseEntity<SearchResponse> search(
            @RequestParam String q) {

        // ③ Service 호출
        SearchResponse result = searchService.search(q);
        return ResponseEntity.ok(result);
    }
}
Service + Repository (DB 접근)
Spring Java 
// ③ Service — 비즈니스 로직 처리
@Service
public class SearchService {

    @Autowired
    private SearchRepository repo;

    public SearchResponse search(String query) {
        // 검색어 정제
        String cleaned = query.trim().toLowerCase();

        // ④ Repository에 DB 조회 요청
        List<SearchResult> items = repo.findByKeyword(cleaned);

        // 결과 가공 & 반환
        return new SearchResponse(items, items.size());
    }
}

// ④ Repository — DB 쿼리 실행 (JPA)
@Repository
public interface SearchRepository
        extends JpaRepository<SearchResult, Long> {

    // 실제 실행 SQL:
    // SELECT * FROM search_results
    // WHERE keyword LIKE '%claude%'
    // ORDER BY rank DESC LIMIT 10
    List<SearchResult> findByKeyword(String keyword);
}
Spring 내부 처리 파이프라인
1
DispatcherServlet — 요청 수신
모든 HTTP 요청의 진입점. 앞단에서 Filter Chain을 통과합니다.
(인증/인가, CORS 처리, 로깅 등이 Filter에서 수행)
Filter → DispatcherServlet
2
HandlerMapping — 컨트롤러 찾기
/search → SearchController.search() 메서드를 찾아줍니다.
@GetMapping, @PostMapping 등의 어노테이션을 보고 매핑합니다.
3
Controller → Service → Repository
MVC 레이어 순서대로 호출. 각 레이어는 역할이 분리되어 있습니다.
Controller: 입출력 / Service: 비즈니스 로직 / Repository: DB 접근
4
DB 쿼리 실행 & 결과 수신
JPA가 SQL을 생성하고 실행합니다. Connection Pool에서 커넥션을 꺼내 쿼리를 날리고, ResultSet을 Java Entity로 변환합니다.
HikariCP (Connection Pool)
5
응답 생성 → 직렬화 → 전송
Java 객체 → JSON (@RestController라면 Jackson이 자동 변환)
또는 HTML (Thymeleaf/JSP 같은 템플릿 엔진이 HTML 생성)
Jackson (Java → JSON 변환)
SSR (Server-Side Rendering) — 서버가 완성된 HTML을 줌

HTML SSR 렌더링 파이프라인
①
완성된 HTML 수신
서버가 DB 결과까지 포함된 완성 HTML을 브라우저로 전송합니다.
②
HTML 파싱 → DOM Tree
브라우저가 HTML을 위에서 아래로 읽으며 DOM 트리를 만듭니다.
<div.result> → Node 생성
③
CSS 파싱 → CSSOM Tree
병렬로 CSS를 파싱해서 스타일 규칙 트리 생성.
④
Render Tree → Layout → Paint
DOM + CSSOM 합체 → 위치/크기 계산 → 픽셀 렌더링 → 화면 표시
✅ 화면 표시 (FCP 달성)
⑤
JS 실행 (선택적)
클릭 이벤트 등 JS가 DOM을 직접 조작할 때만 추가 업데이트 발생.
SSR의 특징
장점
첫 화면이 빠름 (FCP)
서버가 완성된 HTML을 주기 때문에 브라우저가 받자마자 바로 화면에 그릴 수 있습니다.
장점
SEO에 유리
구글 검색봇이 HTML을 바로 읽을 수 있습니다. JS 실행 없이도 콘텐츠가 보입니다.
단점
페이지 이동 시 전체 새로고침
다음 페이지로 이동할 때마다 서버에 새 HTML을 요청해야 합니다.
단점
서버 부하
모든 요청마다 서버가 HTML을 렌더링해야 하므로 트래픽이 많으면 서버가 힘듭니다.
CSR (Client-Side Rendering) — JSON만 받아서 React가 화면 그림
Spring REST → JSON → React 
// Spring이 JSON만 응답 (HTML 아님!)
HTTP/1.1 200 OK
Content-Type: application/json

{
  "total": 3,
  "results": [
    {
      "title": "Claude AI - Anthropic",
      "url":   "https://claude.ai",
      "desc":  "Claude is an AI assistant..."
    }
  ]
}

// React 컴포넌트가 이 JSON을 받아서 화면을 그림
function SearchPage() {
  const [results, setResults] = useState([]);

  useEffect(async () => {
    // API 호출 → JSON 수신
    const res = await fetch('/api/search?q=claude');
    const data = await res.json();

    // State에 저장 → 자동으로 리렌더링!
    setResults(data.results);
  }, []);

  return (
    <div>
      {results.map(r => (
        <ResultCard key={r.url} data={r} />
      ))}
    </div>
  );
}
React CSR 전체 흐름
①
최초: 빈 HTML + JS 번들 수신
처음엔 내용 없는 뼈대 HTML과 React JS 파일(번들)을 다운로드합니다.
<div id="root"></div> + app.bundle.js
②
React 앱 부팅 (Hydration)
JS가 실행되어 React 컴포넌트 트리가 Virtual DOM으로 초기화됩니다.
ReactDOM.createRoot(root).render(<App/>)
③
API 호출 → Spring에서 JSON 수신
useEffect에서 fetch()로 Spring 서버에 JSON 데이터를 요청합니다.
이 동안 화면엔 로딩 스피너가 표시될 수 있습니다.
GET /api/search?q=claude → { results: [...] }
④
setResults() → 리렌더링
JSON을 State에 저장하면 React가 자동으로 리렌더링 트리거합니다.
⑤
Virtual DOM Diffing
이전(빈 배열)과 새(결과 배열) Virtual DOM을 비교 → 추가할 노드 파악
⑥
실제 DOM 업데이트 → 화면 표시
변경된 부분만 실제 DOM에 반영 → 브라우저가 Repaint → ✅ 결과 화면
CSR의 특징
장점
페이지 이동이 빠름 (SPA)
이미 React 앱이 로드되어 있으므로, 이후 페이지 이동은 JSON만 받아서 화면만 바꿉니다. 새로고침 없음!
장점
서버 부하 적음
서버는 JSON 데이터만 주면 됩니다. HTML 렌더링 부담이 없습니다.
단점
첫 화면이 느림 (TTI)
JS 번들 다운로드 → 실행 → API 호출 까지 완료해야 화면이 보임. 초기 로딩이 느립니다.
단점
SEO 불리
처음에 빈 HTML만 있어서 검색 봇이 내용을 못 읽을 수 있습니다. (Next.js SSR로 해결 가능)
SSR(HTML) vs CSR(React) 전체 비교
단계SSR (Spring + HTML)CSR (Spring API + React)
서버 응답 완성된 HTML (데이터 포함) 빈 HTML + JS 번들 → 이후 JSON
첫 화면 빠름 HTML 받으면 바로 렌더링 느림 JS 실행 + API 호출 필요
페이지 이동 느림 매번 서버에 새 HTML 요청 빠름 JSON만 받아 화면 교체 (SPA)
SEO 유리 검색봇이 HTML 바로 읽음 불리 JS 실행 전엔 빈 화면
서버 부하 높음 HTML 렌더링 매번 수행 낮음 JSON만 응답
Spring 역할 Thymeleaf로 HTML 생성 후 반환 @RestController로 JSON만 반환
렌더링 위치 서버 에서 HTML 완성 브라우저 에서 React가 완성
